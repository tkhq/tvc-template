# Deployment Troubleshooting

Common issues encountered during TVC deployment and how to resolve them.

## Container Image Is Private (404 after deployment)

**Symptom:** Deployment shows "Live" but all endpoints return `404 page not found` (plain text, not JSON). The 404 comes from the TVC ingress proxy, not your app.

**Cause:** The container image is not publicly accessible. ghcr.io packages default to PRIVATE. TVC infrastructure cannot pull the image.

**Fix (Option A, recommended): Make the image public.**
1. Go to `https://github.com/users/<USERNAME>/packages/container/<PACKAGE>/settings`
2. Scroll to "Danger Zone", click "Change visibility"
3. Select "Public" and confirm

**Fix (Option B): Provide a pull secret.**
```bash
# Create pull secret JSON
cat > pull-secret.json << EOF
{
  "auths": {
    "ghcr.io": {
      "username": "<GITHUB_USERNAME>",
      "password": "<GITHUB_PAT_WITH_READ_PACKAGES>"
    }
  }
}
EOF

# Deploy with pull secret (CLI encrypts it automatically)
tvc deploy create deploy.json --pivot-pull-secret pull-secret.json
```

**Verify image is public before deploying:**
```bash
TOKEN=$(curl -s "https://ghcr.io/token?scope=repository:<USER>/<REPO>:pull" | jq -r '.token')
STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
  "https://ghcr.io/v2/<USER>/<REPO>/manifests/sha256:<DIGEST>" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Accept: application/vnd.oci.image.manifest.v1+json")
# 200 = public, 401 = private
echo "Image is $([ "$STATUS" = "200" ] && echo "PUBLIC" || echo "PRIVATE")"
```

Note: GitHub's REST API does not support changing package visibility programmatically for user-scoped packages. This must be done through the GitHub UI.

## Wrong App URL Domain

**Symptom:** App deployed successfully but the URL doesn't respond or returns unexpected errors.

**Cause:** The app URL domain depends on which API environment you're authenticated against.

**Fix:** Check `~/.config/turnkey/tvc.config.toml` for the `api_base_url` value:

| api_base_url | App URL Pattern |
|---|---|
| `https://api.turnkey.com` | `https://app-<APP_ID>.turnkey.cloud` |
| `https://api.dev.turnkey.engineering` | `https://app-<APP_ID>.tvc.dev.turnkey.engineering` |

The App ID is the UUID returned by `tvc app create`.

## Config Contains Placeholder Values

**Symptom:** `tvc app create` or `tvc deploy create` fails with "Config file contains placeholder values".

**Cause:** The template generated by `tvc app init` or `tvc deploy init` contains `<FILL_IN_...>` strings that were not replaced.

**Fix:** Replace ALL placeholder values. For app config, fill in `name` and `manifestSetParams.name`. For deploy config, fill in all fields listed in the deployment configuration table in SKILL.md. Also REMOVE the `pivotContainerEncryptedPullSecret` field entirely if the image is public.

## Binary Digest vs Container Digest

**Symptom:** Deployment is rejected or the enclave fails attestation.

**Cause:** Confusion between the container image digest and the binary file digest. These are different values.

- **Container image digest** (used in `pivotContainerImageUrl`): The SHA256 of the OCI image manifest. Shown during `docker push`. Format: `ghcr.io/user/app@sha256:ee5b03eb...`
- **Binary file digest** (used in `expectedPivotDigest`): The SHA256 of the actual binary file inside the container. Computed by extracting the binary and hashing it.

**How to compute the binary digest:**
```bash
docker create --name tmp-extract <IMAGE_URL> /bin/true \
  && docker cp tmp-extract:/<BINARY_NAME> ./binary \
  && docker rm tmp-extract
sha256sum ./binary
```

## Docker Buildx Not Installed

**Symptom:** Container build fails with "BuildKit is enabled but the buildx component is missing".

**Fix:**
```bash
brew install docker-buildx
```

Then add to `~/.docker/config.json`:
```json
{
  "cliPluginsExtraDirs": ["/opt/homebrew/lib/docker/cli-plugins"]
}
```

## Interactive Prompts Blocking Automation

**Symptom:** CLI command hangs waiting for user input in a non-interactive context.

**Fix:** Use the automation flags:
- `--no-input`: Fails instead of prompting (global flag, available on all commands)
- `--yes` / `-y`: Auto-approves all manifest sections (deploy approve)
- Environment variables: Set `TVC_NO_INPUT=true` globally

## Stateful Endpoint Inconsistency

**Symptom:** POST to write data returns success, but subsequent GET to read it returns 404 or different data.

**Cause:** TVC runs multiple enclave instances behind a load balancer. Each instance has independent in-memory state. Sequential requests may hit different instances.

**This is expected behavior, not a bug.** Design your app for stateless, deterministic computation. The enclave's value is the attested computation, not storage. See the "Designing for Enclaves" section in SKILL.md.

## Enclave Provisioning Delay (404 immediately after approval)

**Symptom:** `tvc deploy status` shows "Live" and the image is confirmed public, but the app URL returns `404 page not found` (plain text) immediately after approval.

**Cause:** The enclave needs time to pull the container image, boot, and pass health checks. This is normal and typically takes 1-2 minutes after approval.

**Fix:** Wait and retry. The `404 page not found` from the ingress proxy will be replaced by your app's responses once the enclave is ready. If the 404 persists beyond 5 minutes, investigate other causes below.

## Deployment Status Says "Live" But App Is Not Responding

**Symptom:** `tvc deploy status` shows "Live" but the app URL returns errors persistently (beyond initial provisioning delay).

**Possible causes:**
1. Container image is private (see "Container Image Is Private" above)
2. Wrong URL domain for the environment (see "Wrong App URL Domain" above)
3. Health check configuration mismatch (ensure `healthCheckPort` matches the port your app listens on, and `/health` returns 200)
4. Binary args are wrong (verify `pivotArgs` matches what your app expects, e.g. `["--host", "0.0.0.0", "--port", "3000"]`)
